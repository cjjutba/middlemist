# Logging

Logs in v1 are structured JSON to stdout. Vercel captures them; Sentry receives errors and breadcrumbs through the same logger. There is no log aggregator beyond what Vercel provides; v1 does not need one. The conventions below cover levels, fields, what to log, what to never log, and how to keep logs cheap on hot paths.

## Levels

Four levels: `debug`, `info`, `warn`, `error`.

- **debug:** Local development only. Verbose tracing for the developer working on the feature. Suppressed in production through the log level config.
- **info:** Significant state-affecting events that are not errors. A login succeeded. A cron ran. A rate limit was hit. An audit row was written.
- **warn:** A condition that is not catastrophic but warrants attention. An integration retry. A bounce from Resend. A rate-limit-disabled override active in development.
- **error:** A throw that escaped the wrapper, an integration failure that exhausted retries, or any condition that requires human review.

`AppError` instances raised by the application's business layer are *not* logged at error level (see `docs/engineering/error-handling.md`). They are expected outcomes; logging them at error level would crowd the dashboard.

## Default logger

`pino` for the structured JSON output. It is fast, well-supported, and serializes objects with explicit field control. The wrapper in `src/lib/log.ts` is the single import point for logging across the application.

```typescript
// src/lib/log.ts
import pino from "pino";
import { env } from "@/lib/env";

export const logger = pino({
  level: env.NODE_ENV === "production" ? "info" : "debug",
  base: { env: env.NODE_ENV, app: "middlemist" },
  formatters: {
    level: (label) => ({ level: label }),
  },
  redact: {
    paths: [
      "password",
      "passwordHash",
      "token",
      "secret",
      "authorization",
      "*.password",
      "*.token",
      "*.secret",
      "headers.cookie",
      "headers.authorization",
    ],
    censor: "[redacted]",
  },
  timestamp: pino.stdTimeFunctions.isoTime,
});
```

The `redact` paths cover the most common slipping points. Any field named `password`, `passwordHash`, `token`, `secret`, or `authorization` (at the top level or nested under any key) is replaced with `[redacted]` in the output. `headers.cookie` and `headers.authorization` cover request-shaped log payloads.

Calls look like:

```typescript
import { logger } from "@/lib/log";

logger.info({ userId, proposalId }, "proposal sent");
logger.warn({ provider: "resend", retryCount: 2 }, "send retry");
logger.error({ err, action: "createInvoice" }, "unhandled action error");
```

Pino's call signature puts the structured object first and the message second. Fields are JSON-serializable; `err` is a special field name that pino formats to include the message, stack, and any nested cause.

## Field conventions

A small set of field names appear across most log lines. Sticking to them makes the logs queryable in Vercel.

| Field | Meaning |
|---|---|
| `level` | Log level (set by pino). |
| `time` | ISO timestamp (set by pino). |
| `app` | Always `"middlemist"`. |
| `env` | `development` / `production` / `test`. |
| `requestId` | Generated by middleware, propagated through the request lifecycle. |
| `userId` | The authenticated user, when present. Null on public-route logs. |
| `event` | A short verb naming the operation: `proposal.sent`, `cron.audit-compact`, `rate-limit.hit`. |
| `entityType` / `entityId` | When the log line is about a specific row. |
| `provider` | When the log line is about a third-party (Resend, Upstash, UploadThing). |
| `err` | The error object, formatted by pino's standard serializer. |
| `details` | Free-form structured data specific to the log line. |

```typescript
logger.info(
  {
    requestId,
    userId,
    event: "proposal.sent",
    entityType: "proposal",
    entityId: proposalId,
    details: { recipient: client.email, viaInngest: true },
  },
  "proposal sent"
);
```

The structured fields are how a future query against Vercel logs filters down ("show me every proposal.sent in the last 24 hours") without grep-pattern matching the message string.

## What to log

**Auth events.** `user.signup`, `user.login`, `user.logout`, `user.password-changed`, `user.email-changed`, `user.deletion-requested`, failed-login attempts (at info, with the email and IP). The audit log captures the canonical record; the application log captures the same events for operational debugging.

**State changes.** Already covered by `AuditLog` (in the database), but the application log captures the same events at info level so a failed deploy or a stuck Vercel function shows the in-flight state without a database round-trip.

**Background jobs.** Inngest functions log at start and at end with `event: cron.<name>.start` and `event: cron.<name>.complete`. A failed run logs at error with the throw and the input. Idempotency-critical jobs log the deduplication key.

**Integration calls.** When `sendEmail`, `uploadthing.deleteFiles`, or the FX refresh hits a third-party, the call wraps with a logger entry on success (info) and on failure (warn or error). The provider name is in the `provider` field.

**Rate-limit hits.** Logged at info with `event: rate-limit.hit`, the rule name, the key, and the retry-after. A spike of these is visible in the Vercel logs and indicates either an abusive caller or a UX bug (e.g., a search field firing on every keystroke without debounce).

## What NOT to log

**Plaintext passwords.** Never. The schema redaction covers this, but a developer should not write a log line that includes a password value at all (even if it would be redacted on output).

**Plaintext tokens.** Magic-link tokens, session JWTs, public-link tokens. Hashes are acceptable; values are not. The redaction list covers `token` at any nesting; check before pushing a feature branch that no log line includes a value that would survive redaction.

**Full email body content.** A subject line is OK; the body is too large and may contain client-sensitive content (an invoice amount, a contract scope). Log the recipient address and the template name; that is enough to debug delivery.

**Full file contents.** The URL and metadata are loggable; the bytes are not.

**Customer PII beyond audit needs.** A client's name in a log line that explains why a send failed is acceptable. A client's full address in a log line for an unrelated event is gratuitous; do not include it.

The Sentry integration applies its own scrubbing (see `docs/security/data-protection.md`) for the captured event payloads.

## Performance

Logging in hot paths is expensive. A repository function that runs 30 times per page should not call the logger 30 times. The convention:

- **Page rendering.** No logger calls. The Vercel access log captures the request; that is enough.
- **Repository functions.** No logger calls. The service or action layer is the right place to log "this thing happened."
- **Service functions.** One log line per significant operation. Not per row processed in a loop.
- **Inngest functions.** One log line at start, one at end, and one per failure. Not per item iterated.

For loops, use a counter and log a summary at the end:

```typescript
let processed = 0;
let failed = 0;
for (const invoice of invoices) {
  try {
    await sendReminder(invoice);
    processed += 1;
  } catch (e) {
    failed += 1;
    logger.warn({ invoiceId: invoice.id, err: e }, "reminder send failed");
  }
}
logger.info({ event: "cron.send-reminders.complete", processed, failed }, "reminders done");
```

Per-iteration warn for failures is acceptable because failures are the interesting case; per-iteration info for successes is noise.

## Correlation: requestId

The middleware generates a `requestId` (a `nanoid(12)`) on every incoming request and exposes it through a header (`X-Request-Id`) on the response. The action wrapper reads the same id from the request headers and attaches it to every log line emitted during the action. The id is the join key for "show me everything that happened during this user's click on send."

```typescript
// src/middleware.ts (excerpt)
import { customAlphabet } from "nanoid";
import { NextResponse, type NextRequest } from "next/server";

const generateRequestId = customAlphabet("0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz", 12);

export function middleware(req: NextRequest) {
  const requestId = generateRequestId();
  const res = NextResponse.next();
  res.headers.set("X-Request-Id", requestId);
  return res;
}
```

The action wrapper reads `headers().get("x-request-id")` and includes it in every log line. Sentry breadcrumbs include the same id; a Sentry issue links back to the Vercel logs through the id.

## Reading logs

Two paths.

**Vercel dashboard.** *Project → Logs* shows recent stdout from every function, with search and filter by status code, function name, and free text. The structured JSON format means a search like `level=error` or `event=proposal.sent` filters down quickly.

**`vercel logs`** CLI. `vercel logs <deployment-url> --follow` streams live logs. Useful during a deploy to confirm a new feature is exercising as expected.

There is no Datadog, no Logtail, no log warehouse in v1. Vercel's retention covers the operational window (one hour for free tier, longer on paid). Sentry covers errors with a longer retention. v2 may add an aggregator if the volume warrants.
